# REVENUE_FIRST_OPERATING_RULES.md

## Primary goal
The primary goal of this project is monetization of a language-learning app/product.

The goal is not:
- to build a beautiful app for its own sake
- to collect endless learning content
- to research language markets indefinitely
- to make a generic language-learning tool
- to satisfy every possible learner segment

All research, product design, card writing, localization, audio strategy, PWA work, and Codex tasks must serve the path toward revenue.

## North-star question
For every decision, ask:

> Does this increase the chance that a clearly defined learner, school, agency, employer, or support organization will pay for this product?

If the answer is unclear, the task should be reframed as a small validation experiment or delayed.

## Revenue-first decision hierarchy
Prioritize work in this order:

1. **Identify a buyer or payer**
   - learner
   - language school
   - sending organization
   - registered support organization
   - employer / care facility / factory

2. **Identify a painful, paid problem**
   - passing JFT-Basic is not enough by itself
   - practical work/life Japanese anxiety may be stronger
   -職種別 Japanese may support higher willingness to pay

3. **Create a credible free sample**
   - enough quality to build trust
   - not overbuilt
   - clear target and use case

4. **Make a paid offer hypothesis**
   - paid card pack
   - Japanese audio pack
   -職種別 pack
   - school/agency bundle
   - support-organization training tool

5. **Test demand**
   - landing page response
   - waitlist/signup
   - learner interview
   - direct outreach reply
   - willingness-to-pay signal
   - pilot request
   - pre-order or payment when possible

6. **Only then expand**
   - more cards
   - more languages
   - more audio
   - login/payment/app store
   - advanced progress features

## What counts as progress
Good progress:
- a clearer target segment
- a stronger paid offer
- a usable validation PWA
- direct feedback from target users or organizations
- evidence that one language/country/vertical has demand
- a buyer saying they would pay, pilot, or introduce users
- actual payment or pre-order

Weak progress:
- more features without validation
- large content expansion without a buyer
- polishing UI without conversion purpose
- adding all languages at once
- adding audio before knowing whether it helps sales
- building store/payment infrastructure before offer validation

## Current strategic thesis
The strongest current thesis is:

> Practical Japanese for foreigners preparing to work in Japan, especially JFT-Basic / Specified Skilled Worker / workplace-life Japanese, may be monetizable if packaged as a smartphone-friendly, card-based, localized, audio-ready tool.

This is not yet proven.

## Current likely wedge
Start narrow:
- one target country/language
- one learner level range, roughly A1-A2 / JFT-Basic-ish
- one practical use case: working/living in Japan
- one support-language approach: selected native language + English fallback, or English-only if the segment supports it
- one free PWA sample
- one paid-pack hypothesis

## Guardrails
Do not let the project drift into:
- generic JLPT app
- generic Japanese dictionary
- all-country translation project
- full audio production before demand proof
- app-store-first strategy
- feature-complete PWA before monetization testing

## Codex operating rule
Codex tasks should always state their monetization relevance.

When Codex changes the app or documents, it should update the appropriate handoff/status files and explain how the change supports revenue validation.

## ChatGPT operating rule
ChatGPT should act as revenue strategist, product manager, and critical reviewer.

The assistant should:
- challenge attractive but non-revenue work
- preserve useful owner feedback without blindly following it
- keep the project focused on paid demand
- use current data when market facts matter
- prefer small tests over large speculative builds
- be honest about uncertainty
- never claim monetization is guaranteed

## Success definition
The project succeeds when it produces evidence of real willingness to pay and then converts that evidence into a paid product, pilot, pre-order, subscription, bundle, or B2B sale.

A working app alone is not success.
A large deck alone is not success.
A beautiful UI alone is not success.

Revenue or credible paid-demand evidence is the target.
